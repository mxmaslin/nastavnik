# Тестовое задание для fullstack

## Формулировка задания для кандидата

### Задание

Реализовать простой чат с пользователем, имитирующем проведение урока. 
Нужно подготовить объект - список вопросов с правильными ответами. Создай этот объект сам
Система задает пользователю вопрос(выбирается из списка вопросов по порядку) и принимает от пользователя ответ.
Эта пара вопрос - ответ отправляются в ml-сервис. в нем реализован эндпоинт fast-api с функцией-заглушкой, которая проверяет правильный ли ответ дал пользователь. 
Эта заглушка имитирует обращение к llm - запрос к llm занимает не менее 5 секунд. Эта функция возвращает 1 или 0, где 1 - это корректный ответ.
Если пользователь не давал ответа больше 30 секунд, то система записывает пустой ответ.
В 1 случае из 3 функция-заглушкаа возвращает код ошибки(llm-недоступна).
 
Вопрос-ответ - это пример учебного взаимодействия, записи о которых мы храним в таблице interaction_records в БД.
Нам очень важен порядок, корректность, какими были вопросы и ответы и дата и время взаимодействия.
Нам очень важно, чтобы все записи обязательно попадали в таблицу, даже такие, когда пользователь не ответил и закрыл урок или мы не смогли проверить корректность ответа.
Также нам важно, чтобы пользователь видел, что система рабоатет и что-то делает, пока он ждет следующий вопрос.

## Что было навайбкодено

### Описание постановки

Реализован обучающий сервис, имитирующий проведение "урока", для которого заранее подготовлен упорядоченный список ответов. Многопользовательность не реализована т.к. не требовалась по ТЗ.

Пользователю через веб-интерфейс (Next.js) предлагается ознакомиться с текстом, после ознакомления - ответить на вопросы по этому тексту. Логика получения и обработки пользовательского ввода реализована на Django, DRF. Каждый ответ пользователя обрабатывается Celery-таской. Логика таски состоит в том, что на эндпоинт "ML-сервиса" направляется запрос, содержащий ID текущего вопроса пользователю, а также сам ответ пользователя. "ML-сервис" работает на FastApi и представляет собой асинхронную функцию-заглушку, которая, получив ID вопроса и ответ пользователя, "проверяет" правильность ответа. Правильность проверяется не за счёт обработки ответа ИИ-шкой (её в требуемом стеке нет), а путём проверки ответа пользователя в кэше Redis, и, в случае cache miss, асинхронного запроса в PostgreSQL через SQLAlchemy ORM. В этом запросе мы сверяем в БД ответ пользователя с правильным ответом. Ответы пользователя, вне зависимости от их правильности / неправильности, сохраняются в БД в таблицу "interaction_record" вкупе с датой и временем ответа на вопрос урока.

Если пользователь не давал ответа больше 30 секунд (во время которых отображается уменьшающийся прогрес-бар этого тридцатисекундного периода ожидания), то сервис записывает пустой ответ. В одном случае из трёх функция-заглушка возвращает HTTP-код ошибки ("LLM недоступна"). Пока идёт фоновая проверка ответа в ML-сервисе, взаимодействие сервиса с пользователем не прерывается: он в любой момент может завершить урок. 

Статистика: после завершения текущего урока показываются метрики **только по этому уроку**; при открытии раздела статистики **с главного экрана** (без активного урока) — **по всем ранее пройденным** урокам в рамках того же `session_id` в браузере. Повторный старт того же урока после завершения на backend сбрасывает запись ответов по этой паре "сессия + урок".

Реализованы rate-лимиты от DDoS, код обучающего сервиса покрыт unit- и интеграционными тестами, сервисы развёрнуты через Docker Compose.

### Использованный стек

- Веб-интерфейс: Next.js
- Сервис приёма и обработки вопросов пользователя: Django, DRF, Celery, Redis, PostgreSQL
- "ML-сервис": FastAPI, Redis, PostgreSQL
- Тесты: Pytest
- Очередь: RabbitMQ
- Деплой: Docker Compose, Github CI
- Метрики: Prometheus, Grafana

### Запуск и просмотр

В корне проекта:

```bash
make up
```

Подождать, пока `backend` станет healthy (миграции и `seed_data` выполняются при старте контейнера).

| Сервис | URL |
|--------|-----|
| Веб-интерфейс (Next.js) | http://localhost:3000 |
| Django / DRF | http://localhost:8000 |
| ML-сервис | http://localhost:8001 |
| Prometheus | http://localhost:9090 |
| Grafana (admin / admin) | http://localhost:3001 |
| RabbitMQ management | http://localhost:15672 (guest / guest) |

Интерфейс открывайте по **http://localhost:3000**. Запросы к API идут с браузера на **http://localhost:8000** (CORS в Django разрешён). При необходимости базу API можно задать переменной **NEXT_PUBLIC_API_URL**.

**OpenAPI / Swagger UI:** **http://localhost:8000/api/schema/swagger-ui/** (схема YAML: `/api/schema/`). Вопросы в API отдаются с полем **`choices`** (три варианта для радиокнопок на фронте).

В проде `correct_answer` и поля дистракторов в JSON не отдаются; клиент шлёт выбранную строку в `POST /api/answer/submit/` как раньше.

Тесты:

```bash
make test
```

**E2E (Playwright)** — в отдельном job в GitHub Actions (`e2e`). Локально: поднять стек с backend (**`make up`** или как минимум сервисы до API на порту 8000), затем:

```bash
make test-e2e
```

(при первом запуске ставит браузер Chromium для Playwright).

**Интеграционные тесты в репозитории:** у **backend** это pytest с **реальной БД** и HTTP к **DRF** (`APIClient` / запросы к `/api/...`) — это интеграционные API-тесты, не браузер. У **ml_service** — pytest с отдельными проверками сервиса. Сквозные сценарии «браузер + Next + API» покрываются **Playwright E2E**.
